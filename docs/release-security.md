# Release integrity and provenance

Trove's `v1.0.0` supply-chain policy applies to every tagged release starting
with `v0.16.0`:

- GitHub Actions and release tools are pinned to immutable commits or exact
  versions.
- Every binary archive has a SHA-256 entry in `checksums.txt` and a matching
  SPDX JSON software bill of materials (SBOM) generated by Syft.
- Every multi-platform container image is published as one OCI index for
  `linux/amd64` and `linux/arm64`, with BuildKit provenance and an SPDX SBOM
  attached to the index.
- GitHub keyless artifact attestations cover the checksummed release files and
  the digest-pinned container images. This is the signing mechanism: Trove does
  not maintain a long-lived private release-signing key.
- The release job refuses to treat an existing GitHub release as complete when
  an archive, SBOM, checksum file, or image-digest file is missing.

This keeps the trust root in the public repository, tagged workflow, and
GitHub's OIDC-backed attestation identity instead of a maintainer-held secret.

## Verify a binary archive

Download the archive, its `.sbom.spdx.json` file, and `checksums.txt` from the
same release. Then verify both the checksum and GitHub attestation:

```sh
grep 'trove-server_.*_linux_amd64.tar.gz' checksums.txt | sha256sum --check
gh attestation verify trove-server_0.16.0_linux_amd64.tar.gz \
  --repo techdox/trove
```

Use `shasum -a 256 --check` instead of `sha256sum --check` on macOS.

## Verify a container image

Pin deployments to the immutable digest shown in `digests.txt`, then verify the
attestation for the published OCI image:

```sh
gh attestation verify oci://ghcr.io/techdox/trove-server:0.16.0 \
  --repo techdox/trove
```

Tags remain convenient discovery names; production deployments should use the
recorded digest when they need a cryptographically fixed artifact.
